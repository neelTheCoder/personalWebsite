Why General-Purpose BCIs are the Next Interface Paradigm

April 2026 · 10 min read

Every major interface shift — from command line to GUI, from desktop to mobile — followed the same arc: a technology that seemed niche became infrastructure. Brain-computer interfaces are next in line, and the question isn't whether they'll arrive but what shape they'll take when they do.

The problem with vertical BCIs

Most BCI companies today are built around a single application: restoring motor function, treating depression, enabling communication for ALS patients. This is important work. But it creates an ecosystem of siloed, application-specific systems that don't compose — a user can't take signal processing learned in one context and apply it to another.

General-purpose BCIs ask a different question: what if we built the underlying platform first, and let applications proliferate on top? This is the bet that made Android and iOS so powerful — not the apps themselves, but the substrate that made apps possible.

What "general purpose" actually means

A general-purpose BCI needs to do a few things well. It needs a hardware-agnostic signal processing layer that abstracts over EEG, fNIRS, and emerging modalities. It needs a low-latency decoding pipeline that can be tuned for different cognitive signals without retraining from scratch. And it needs an application interface — something like an OS API — that lets developers build BCI-native experiences without needing neuroscience PhDs.

Numa is an attempt at this substrate. We're not building the headset. We're building the software that makes the headset useful for more than one thing.

The timing question

Hardware isn't there yet — consumer-grade EEG is still noisy, fNIRS is awkward to wear, and invasive BCIs require surgery. But the software layer can be built now. The bottleneck isn't the signal quality of today's hardware; it's the absence of development infrastructure that would let people experiment with tomorrow's hardware cheaply. That's the gap worth filling.

Benchmarking AI Agents at Scale: What We Got Wrong

March 2026 · 8 min read

When we set out to benchmark four AI coding agents across 350 real-world workflow tasks, we thought the hard part would be the infrastructure. We were wrong. The hard part was defining what "good" meant.

The rubric problem

Early in the project, we tried to evaluate agents on binary pass/fail: did the task complete or not? It seemed clean. It was useless. A task like "refactor this module to improve readability" doesn't have a binary answer. Two humans given the same output would disagree on whether it passed.

We ended up building a rubric-based LLM judging layer — a meta-model that evaluates agent outputs against structured criteria. This introduced its own problems: rubric quality, judge consistency, and the uncomfortable question of whether you're measuring the task or measuring the rubric.

What delivery rate actually tells you

The most useful metric we found wasn't accuracy — it was delivery rate: how often does the agent produce any coherent output at all? Agents that fail silently are more dangerous than agents that fail loudly. Measuring the shape of failure turned out to be more informative than measuring the rate of success.

The Placebo Effect in Machine Learning Evaluation

February 2026 · 6 min read

In medicine, the placebo effect is well understood: the act of treatment produces real outcomes independent of the treatment itself. In machine learning evaluation, there's an analogous phenomenon — and it's less well-studied.

When the benchmark shapes the model

Once a benchmark becomes widely used, models are optimized against it. The benchmark stops measuring the capability it was designed to measure and starts measuring benchmark performance. This is Goodhart's Law, but in ML it operates faster and at larger scale than anyone anticipated.

The fix isn't to build better benchmarks — it's to build benchmarks that are inherently hard to optimize against without actually improving the capability. Real-world workflow tasks, evaluated by rubric-based judges with high variance, are one approach. The goal is evaluation that can't be gamed, because gaming it would require actually solving the problem.
